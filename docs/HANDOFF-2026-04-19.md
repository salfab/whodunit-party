## Ce Qui A Ete Fait

### Gameplay et roles

- La liste des suspects cote ecran enqueteur est melangee avant d'etre renvoyee.
- Les noms de personnages qui spoilent le role ont ete remplaces dans les packs et dans des migrations de correction.
- Dans la fiche personnage:
  - le libelle `Alibi` a ete remplace par une aide plus douce;
  - le `dark_secret` n'est plus affiche pendant la partie;
  - il n'est revele qu'apres accusation / aveu.
- Le coupable n'est plus defini par le pack.
- Le coupable est choisi par HMAC SHA-256 deterministe, avec `JWT_SECRET`, a partir de:

```text
sessionId:mysteryId:sortedSuspectPlayerIds:v1
```

Raison: `Math.random()` pourrait changer le coupable entre chargement, accusation, revelation ou redemarrage serveur. Le HMAC donne un hasard stable sans colonne DB et les clients ne peuvent pas recalculer le coupable sans le secret serveur.

### Migration DB acceptee finalement

Decision initiale: pas de migration DB. Cette decision a change.

Nouvelle decision: la DB utilise aussi `investigator | suspect`.

Migration ajoutee:

- `supabase/migrations/20260419002000_migrate_character_sheet_roles_to_suspect.sql`

Elle fait:

- renomme l'ancien enum `player_role` en `player_role_legacy`;
- cree `player_role AS ENUM ('investigator', 'suspect')`;
- convertit toute ancienne valeur non-investigator (`guilty`, `innocent`) en `suspect`;
- supprime l'ancien enum.

Important:

- `character_sheets.role` ne doit plus stocker `guilty` ou `innocent`.
- `guilty` / `innocent` restent uniquement des `assignedRole` runtime pour les joueurs pendant une manche.
- Les types Supabase dans `src/types/database.ts` ont ete mis a jour manuellement pour `player_role: "investigator" | "suspect"`.

## Ce Qui Reste A Faire

### 1. Faire remplir le template par un autre LLM

Fichier a fournir a Claude / ChatGPT:

- `seed-data/confession-rewrite-template.json`

Consigne recommandee:

- remplir uniquement `character_sheets[].dark_secret` pour les fiches `role: "suspect"`;
- ne pas modifier `zip_file`, `mystery_index`, `rewrite_key`, `title`, `role`, `character_name`, `occupation`, `alibi`;
- ecrire un aveu narratif complet, coherent avec:
  - la trame du mystere;
  - le ton du pack;
  - le personnage;
  - son alibi;
  - la possibilite que ce personnage soit le vrai coupable;
- le texte doit pouvoir etre lu tel quel au moment ou le coupable avoue;
- eviter les formulations repetitives entre personnages;
- ne pas mentionner les mecanismes du jeu.

### 2. Creer le script de reimport

Il faut encore ecrire un script, probablement:

```text
scripts/import-confession-rewrite-template.mjs
```

Comportement attendu:

- lire un JSON rempli, par exemple `seed-data/confession-rewrite-template.json` ou un fichier donne en argument;
- pour chaque entree:
  - ouvrir `seed-data/mysteries/{zip_file}`;
  - lire `mystery.json`;
  - retrouver le mystere par `mystery_index`;
  - retrouver chaque fiche par `rewrite_key`;
  - remplacer uniquement `dark_secret` des fiches `role: "suspect"`;
  - ne pas toucher les investigateurs;
  - conserver les autres champs du ZIP actuel;
  - reecrire le ZIP.

Validation minimale dans ce script:

- tous les `zip_file` existent;
- tous les `mystery_index` sont valides;
- tous les `rewrite_key` existent dans le mystere cible;
- chaque suspect a un `dark_secret` non vide;
- aucun `dark_secret` suspect n'est encore `""`;
- aucun role autre que `investigator | suspect` dans les ZIPs;
- le JSON final de chaque ZIP valide le schema.

## Rappels De Conception

- Le pack ne definit jamais le coupable.
- Le role de fiche stocke est seulement:

```text
investigator | suspect
```

- Le role de manche runtime est:

```text
investigator | guilty | innocent
```

- `guilty` et `innocent` ne doivent plus etre stockes dans `character_sheets.role`.
- `guilty` / `innocent` peuvent encore apparaitre:
  - dans les messages UI runtime;
  - dans `AssignedRole`;
  - dans des tests d'import legacy;
  - dans de vieilles migrations avant la migration finale.
